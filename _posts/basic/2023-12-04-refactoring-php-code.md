오늘은 회사내 프로젝트에서 PHP 코드를 리팩토링하게 된 과정과 결과에 대해서 기록한다. 
특히, 데이터 처리가 많이 필요한 시스템에서 성능을 어떻게 개선했는지에 한번 집중해서 적어보려 한다. 

**리팩토링을 하게 된 배경**

사실 이 시스템이 엄청나게 큰 건 아님, 그런데 운영팀에서 데이터 처리가 너무 느려서 사용하기 어렵다고 연락이옴😭. 
300명 분량의 데이터를 처리할 때마다 1건당 1분씩 걸리니까, 300명이면 대략 300분이 필요(OMG). 
그래서 이 문제를 해결하기 위해 리팩토링을 시작함.

**왜 이렇게 느렸을까?**

가장 큰 이유는 데이터 처리 과정이 복잡했기 때문. 예를 들어, 기존에는 `select id, member_info from 고객테이블 where id = ‘유저1’` 이런 식으로 많은 검증 과정을 거치면서 사람을 조회했는데. 이 방식으로 조회하면 조회가 아무리 빨라도 n * n log n 식으로 시간이 증가. 
당연히!! 이렇게되면 유저 수가 많아지면 자연스럽게 느려질 수밖에 없지.

**기존 코드**

```php
// 기존의 PHP 코드 일부
include_once("./_common.php");
// ... 나머지 코드 ...

for($i = 0; $i < count($list); $i++){
    // 여러 단계를 거쳐 데이터 검증하고 조회하는 로직
    // ... 코드 ...
}

// ... 나머지 코드 ...
```

이 코드는 여러 단계를 거쳐 데이터를 검증하고 조회하는 과정으로 되어 있음
이런 방식으로 코딩하게 되면 데이터 처리에 많은 시간이 소요되는데 DB 크기가 커지면 커질 수로 n2 만큼 커짐.

```text
그러니까 예를 들어 이진 트리로 탐색한다고 했을때 컴퓨터에서 소화 가능한 비트가 최대 2^63일텐데
최대 63번은 안넘어간다고 가정해보자고    
처음 데이터가 별로 없을때 하나의 테이블의 데이터를 탐색할때   
1만개 정도였다 치자 14번정도면 탐색할텐데   
조회할 테이블 조인할 테이블이 많아지면   
쿼리 하나 조회할때 5개 테이블이 사용되면    
14 * 14 * 14 * 14 * 14 = 537,824   
이렇게 10번정도 테이블이 조회되면   
5,378,240   
데이터가 100만 정도로 불어나 버리면     
20번이면 다 탐색 가능할텐데   
20 * 20 * 20 * 20 * 20 = 3,200,000   
이렇게 10번정도 테이블이 조회되면   
32,000,000
차이가 약6배 정도 나버린다고  
```

주요 변경 사항은 데이터 처리 과정을 최적화하고, 특히 `SELECT` 쿼리와 `INSERT` 작업을 개선하는 데 집중.    
여기 개선된 코드의 주요 부분

**1. 대량 데이터 처리 최적화**

- **Bulk Insert 활용**: 대량의 데이터를 한 번에 데이터베이스에 삽입하기 위해 `bulk_insert` 함수를 사용. 이 방식은 개별 `INSERT` 문을 여러 번 실행하는 것보다 훨씬 효율적임

```php
function bulk_insert($tableName, $lst_insert_col, $lst_insert_val) {
    // 코드 구현...
}
```

- **성능 개선을 위한 데이터 조회 최적화**: 대량의 유저를 한 번에 조회하기 위해 `IN` 절을 사용. 이 방식은 개별적으로 여러 번 조회하는 것보다 훨~~~씬 빠르고 효율적.    
- 당연히 매번 조회하는 것보다 메모리에서 꺼내다 쓰는게 빠름

```php
$sql = "SELECT * FROM 등록정보테이블 WHERE id IN ('유저1', '유저2', ..., '유저300')";
```
조회후 데이터는 map 구조를 가진 오브젝트에 넣어서 두번 다시 조회하지 않도록 했지
원래는 300번 조회해야 하는걸 1번으로 줄인거지 이런게 여러개니까 300번 조회해야 로직이 해당 검증코드안에 10개가 들어가있으면 300 * 10번 = 3,000번 인데 이거를 10번이면 되도록 만들었어

**2. 코드 모듈화 및 가독성 향상**

- **함수 분리**: 코드의 가독성과 유지보수를 향상시키기 위해 복잡한 로직을 작은 함수로 분리했어. 예를 들어, `bulk_insert_lms_score`와 `bulk_insert_attend_hist` 함수는 특정 작업을 수행하는 데 집중하고 있어.

**3. 에러 핸들링과 데이터 무결성 보장**

- **트랜잭션 관리**: 데이터의 일관성과 무결성을 유지하기 위해 트랜잭션을 사용했어. 모든 데이터 삽입이 성공하면 커밋(commit)되고, 하나라도 실패하면 롤백(rollback)되도록 구현.
  _이전코드는 한명 한명 데이터를 넣고 커밋한다음 혹시나 중간에 멈추면 이미 들어간 데이터는 제외하고 다른 사람을 넣는 식으로 되어 있더라고 이런방식은 중간에 언제 멈출지도 모르고 문제가 많아
  원래 시스템 자체가 느렸기 때문에 해당 개발자가 자구책으로 생각한 모양인데 이렇게 하면 데이터가 많아지면 많아질수록 느려지는 구조야_

```php
sql_begin_transaction();
try {
    // Bulk insert operations...
} catch (Exception $e) {
    sql_query("ROLLBACK");
    // Error handling...
}
```

**4. 보안 강화**

- **사용자 입력 데이터 검증 강화**: 엑셀로 입력되는 데이터기 때문에 데이터 검증이 중요해 반드시 거쳐야 하고 여러 테이블의 데이터를 가져오기 때문에 검증로직이 매우 중요해 때문에 사용자 입력 데이터에 대한 검증 로직을 추가.
